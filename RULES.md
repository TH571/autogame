## 核心规则

### 1. 基本组队要求

| 规则 | 说明 |
|------|------|
| **最少人数** | 至少 4 人才能组队（可配置） |
| **种子选手** | 种子选手必须参与每场活动（可配置） |
| **关联用户** | 只有活动的关联用户才能参与组队 |
| **时间段** | 只计算下午 (14:00-18:00) 和晚上 (19:00-22:00) 两个时间段 |

### 2. 用户优先级

组队时按以下优先级选择用户：

1. **从未参与的用户** - 最高优先级（保证至少参与一次）
2. **参与次数少的用户** - 参与次数 × 10 作为基础分
3. **最近未参与的用户** - 间隔天数越多，优先级越高
4. **前一天未参与的用户** - 避免连续参与，惩罚 500 分

### 3. 公平分配原则

- 参与次数少的用户优先
- 避免用户连续参与（前一天参与过则优先级降低）
- 保证每位用户至少参与一次（如可能）

## 组队流程

### 流程图

```
开始组队
    ↓
获取种子选手
    ↓
遍历未来 14 天
    ↓
对于每一天：
  ├─ 下午 (timeSlot=1)
  │   ├─ 获取可用关联用户
  │   ├─ 检查 ≥4 人？
  │   ├─ 检查种子选手有空？
  │   └─ 选择 4 人组队 → 创建活动
  │
  └─ 晚上 (timeSlot=2)
      ├─ 获取可用关联用户
      ├─ 检查下午参与人员
      ├─ 优先选择下午参与的用户
      ├─ 检查 ≥4 人？
      ├─ 检查种子选手有空？
      └─ 选择 4 人组队 → 创建活动
```

### 详细步骤

1. **获取种子选手**
   - 查找 `is_seed=1` 的用户
   - 如果没有种子选手，组队失败

2. **遍历未来 14 天**
   - 每天检查两个时间段：下午 (1) 和晚上 (2)
   - 不再支持"下午 + 晚上" (3) 的时间段

3. **检查可用用户**
   - 获取该时间段已申报的用户
   - 只包括活动的关联用户
   - 排除非关联人员

4. **人员一致性检查（晚上）**
   - 如果下午已有组队，记录参与人员 ID
   - 晚上组队时优先选择下午参与的用户
   - 保持下午和晚上人员的一致性

5. **创建活动和记录**
   - 创建活动记录
   - 添加成员
   - 记录参与历史

```javascript
  } else {

  }

```

### 示例场景

**场景 1：下午参与人员晚上都有空**
```

组队结果：蚊子、铁、李娜、刘洋（下午 2 人 + 补充 2 人）
```

## 活动规则配置

### 可配置参数

| 参数 | 字段 | 默认值 | 说明 |
|------|------|--------|------|
| **最少人数** | `min_players` | 4 | 达到此人数即可组队 |
| **最多人数** | `max_players` | 4 | 每队最多人数 |
| **每局人数** | `players_per_game` | 4 | 参与人数必须为此数的整数倍 |
| **需要种子** | `require_seed` | true | 只有种子选手有空才组织活动 |
| **种子强制** | `seed_required` | true | 种子选手必须参加每场活动 |

### 规则组合示例

#### 标准模式（默认）
```json
{
  "min_players": 4,
  "max_players": 4,
  "players_per_game": 4,
  "require_seed": true,
  "seed_required": true
}
```
- 种子选手必须参加
- 4 人成队

#### 自由模式
```json
{
  "min_players": 4,
  "max_players": 4,
  "players_per_game": 4,
  "require_seed": false,
  "seed_required": false
}
```
- 不需要种子选手
- 任意 4 人即可组队

#### 指导模式
```json
{
  "min_players": 4,
  "max_players": 4,
  "players_per_game": 4,
  "require_seed": true,
  "seed_required": false
}
```
- 种子选手有空才组织活动
- 但种子选手不强制参与每场

## 时间锁定机制

### 24 小时后悔期

| 时间段 | 规则 |
|--------|------|
| **申报后 24 小时内** | ✅ 可随时修改或删除 |
| **超过 24 小时** | 🔒 锁定 3 天内的时间（保证团队稳定性） |
| **3 天后** | 📅 始终可以修改 |

### 锁定检查逻辑

```javascript
async canModify(userId, date, timeSlot) {
  const existing = await checkAvailability(userId, date, timeSlot);
  
  if (!existing) {
    return { canModify: true };
  }
  
  const hoursSinceLastModified = (now - lastModified) / (1000 * 60 * 60);
  
  // 24 小时后悔期内，可以修改
  if (hoursSinceLastModified < 24) {
    return { canModify: true, reason: 'regret_period' };
  }
  
  // 24 小时后，只能修改 3 天后的日期
  if (inputDate >= threeDaysLater) {
    return { canModify: true, reason: 'future_date' };
  }
  
  return { canModify: false, reason: 'locked' };
}
```
